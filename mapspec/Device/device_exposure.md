# Device / DeviceUseStatement → device_exposure

OMOP CDM v5.4. The `device_exposure` table records a person's exposure to a foreign physical object or instrument. One FHIR DeviceUseStatement (with its referenced Device) maps to one `device_exposure` row. Alternatively, one `Procedure.usedCode` coding entry maps to one `device_exposure` row.

## Field Mapping

| FHIR Path | OMOP Field | Type | Required | Notes |
|---|---|---|---|---|
| (generated) | `device_exposure_id` | integer | Yes (PK) | Surrogate key. Autogenerated sequence or hash of FHIR resource ID. |
| `DeviceUseStatement.subject` | `person_id` | Reference -> integer (FK PERSON) | Yes | Resolve Patient reference to integer `person_id`. DeviceUseStatement.subject is 1..1 and restricted to Patient. |
| `DeviceUseStatement.device` -> `Device.type` | `device_concept_id` | CodeableConcept -> integer (FK CONCEPT) | Yes | Resolve `Device.type.coding` to a Standard Concept in the Device domain via vocabulary lookup. | `DeviceUseStatement.timing[x]` (start) | `device_exposure_start_date` | dateTime/Period -> date | Yes | If `timingPeriod`: use `Period.start`. If `timingDateTime`: use that value. If `timingTiming`: use the overall bounds start or first event. Fallback to `recordedOn` if timing is absent. |

### Alternative mapping: Procedure.usedCode

When the source is `Procedure.usedCode` rather than DeviceUseStatement + Device:

| FHIR Path | OMOP Field | Notes |
|---|---|---|
| `Procedure.usedCode.coding` | `device_concept_id` | Vocabulary lookup of the device code |
| `Procedure.usedCode.coding.code` | `device_source_value` | Verbatim code |
| `Procedure.subject` | `person_id` | Patient reference from the parent Procedure |
| `Procedure.performed[x]` (start) | `device_exposure_start_date` | From the Procedure timing |
| `Procedure.performed[x]` (end) | `device_exposure_end_date` | From the Procedure timing |
| `Procedure.encounter` | `visit_occurrence_id` | Direct encounter reference |

This approach is used by ETL-German and HealthcareLakeETL. It loses UDI, lot/serial, and device-specific metadata but avoids the two-resource join problem.

FHIR fields with no OMOP target (lost unless using extensions): `Device.manufacturer`, `Device.manufactureDate`, `Device.expirationDate`, `Device.modelNumber`, `Device.partNumber`, `Device.deviceName`, `Device.specialization`, `Device.version`, `Device.property`, `Device.safety`, `Device.url`, `Device.contact`, `Device.owner`, `Device.location`, `Device.parent`, `DeviceUseStatement.identifier`, `DeviceUseStatement.basedOn`, `DeviceUseStatement.derivedFrom`, `DeviceUseStatement.bodySite`, `DeviceUseStatement.reasonCode`, `DeviceUseStatement.reasonReference`, `DeviceUseStatement.note`.

## Vocabulary Mappings

### Device type (`Device.type` → `device_concept_id`)

FHIR `Device.type` is a CodeableConcept, typically coded in SNOMED CT (device hierarchy, descendant of 49062001 "Device"). The OMOP CONCEPT table includes SNOMED device concepts in the "Device" domain.

| Source vocabulary | FHIR system URI | OMOP vocabulary_id | Mapping strategy |
|---|---|---|---|
| SNOMED CT | `http://snomed.info/sct` | SNOMED | Direct lookup in CONCEPT table; filter to domain_id = 'Device'. Standard concepts map directly. |
| GMDN | `https://www.gmdnagency.org` | GMDN | Requires source_to_concept_map or custom vocabulary load. Not in default OMOP vocabulary bundle. |
| GUDID / UDI | `http://hl7.org/fhir/NamingSystem/fda-udi` | None standard | UDI-DI is an identifier, not a coded concept. Store in `unique_device_id`, not `device_concept_id`. |
| ICD-10-PCS | `http://www.cms.gov/Medicare/Coding/ICD10` | ICD10PCS | May appear in Procedure context. Map via standard_concept relationships. |
| Local/custom | varies | varies | Use `device_source_value` and set `device_concept_id = 0`. |

Common SNOMED device concept examples:

| SNOMED code | Display | OMOP concept_id | Notes |
|---|---|---|---|
| 360046005 | Cardiac pacemaker | 4141267 | Standard, Device domain |
| 304120007 | Total hip replacement prosthesis | 4216669 | Standard, Device domain |
| 303608005 | Ophthalmic intraocular lens prosthesis | 4230167 | Standard, Device domain |
| 272265001 | Bone screw | 4109487 | Standard, Device domain |
| 14106009 | Cardiac defibrillator | 4324032 | Standard, Device domain |

When `Device.type` is absent or the code cannot be mapped: `device_concept_id = 0`, store the verbatim in `device_source_value`.

### Type concept (`device_type_concept_id`)

| Provenance | OMOP concept_id | concept_name | Used by |
|---|---|---|---|
| EHR record | 32817 | EHR | ETL-German (CONCEPT_EHR) |

## Reference Resolution

### `DeviceUseStatement.subject` → `person_id`

`DeviceUseStatement.subject` is a required (1..1) reference to Patient. Resolution is straightforward:

1. Extract the Patient reference (`Patient/{id}` or `urn:uuid:{id}`).
2. Resolve to integer `person_id` via the Patient mapper's lookup (sequence, hash, or table).
3. If the Patient has not been processed, defer the DeviceUseStatement or fail.

omoponfhir validates that the subject resource type is "Patient" and throws `FHIRException` if not (line 275-282 in `OmopDeviceUseStatement.java`).

### `DeviceUseStatement.device` → Device resource

`DeviceUseStatement.device` is a required (1..1) reference to Device. Two resolution patterns:

1. **Contained Device.** omoponfhir embeds Device as a contained resource within DeviceUseStatement. The reference is local (`#id`). The mapper iterates `contained[]` to find the matching Device (lines 250-258).
2. **External reference.** The reference points to a standalone Device resource. The mapper must fetch or pre-cache the Device. This requires the Device to be processed before or alongside the DeviceUseStatement.

From the resolved Device, extract:
- `Device.type` -> `device_concept_id` / `device_source_value`
- `Device.udiCarrier[0].deviceIdentifier` -> `unique_device_id`
- `Device.lotNumber` / `Device.serialNumber` -> `production_id`

### `DeviceUseStatement.source` → `provider_id`

`DeviceUseStatement.source` (0..1) references Patient, Practitioner, PractitionerRole, or RelatedPerson. OMOP `provider` corresponds to Practitioner only. Strategy:

1. If reference is to Practitioner: resolve to integer `provider_id`.
2. If reference is to PractitionerRole: resolve to the underlying Practitioner.
3. If reference is to Patient or RelatedPerson: cannot map to OMOP provider; skip or log warning.
4. If absent: `provider_id = null`.

omoponfhir resolves source as Practitioner (lines 315-326).

### Visit occurrence (no direct reference)

DeviceUseStatement does not have a direct encounter/visit reference. Strategies to populate `visit_occurrence_id`:

1. **Via derivedFrom.** `DeviceUseStatement.derivedFrom` may reference a ServiceRequest or Procedure that has an encounter reference. Traverse the chain.
2. **Via temporal overlap.** Match `device_exposure_start_date` against the patient's `visit_occurrence` records by date range.
3. **Via Procedure context.** If the device_exposure originates from `Procedure.usedCode`, the Procedure has a direct `encounter` reference (ETL-German approach, line 249).
4. **Leave null.** If no visit can be determined. This is acceptable per OMOP (field is optional).

## Edge Cases

| Case | Handling |
|---|---|
| Missing `Device.type` | `device_concept_id = 0`, `device_source_value = null`. Log warning. The record is still valid -- OMOP allows concept 0. |
| `timing[x]` is `dateTime` (not Period) | Use the dateTime for `device_exposure_start_date`. `device_exposure_end_date = null` (point-in-time exposure). |
| `timing[x]` is `Timing` (repeating) | No natural OMOP representation. Options: (a) use `Timing.repeat.boundsPeriod` as start/end, (b) create one row per event occurrence, (c) skip and log warning. No implementation handles this case. |
| `timing[x]` absent | Fall back to `DeviceUseStatement.recordedOn` for start date. If that is also absent, the record cannot be created (`device_exposure_start_date` is NOT NULL). Skip and log. |
| Multiple `Device.udiCarrier[]` entries | Take the first entry's `deviceIdentifier` for `unique_device_id`. OMOP has a single field; additional UDI carriers are lost. |
| `DeviceUseStatement.status = entered-in-error` | Do not create a device_exposure row. ETL-German filters on acceptable status values (`completed`, `active`, `on-hold`). |
| Device is external (not contained) | Must resolve the Device reference before mapping. If the Device resource is unavailable, log warning and set `device_concept_id = 0`, `unique_device_id = null`. |
| `Procedure.usedCode` with multiple codings | Create one device_exposure row per coding entry (ETL-German approach, lines 232-263). Deduplicate if needed. |
| `device_source_value` exceeds 50 chars | OMOP field is varchar(50). Truncate the source value. `system:code` format from omoponfhir may exceed this limit for long system URIs. |
| `Device.type` has multiple codings (e.g., SNOMED + GMDN) | Use the first coding that maps to a standard OMOP concept. Store all codings in `device_source_value` or pick the primary one. omoponfhir uses `getCodingFirstRep()`. |
| Implantable device with no end date | `device_exposure_end_date = null`. This is correct for permanent implants -- the exposure is ongoing. |
| `production_id` composition | OMOP v5.4 added this field. Combine `Device.lotNumber` and/or `Device.serialNumber`. No standard format. Suggestion: `lot:{value}` or `serial:{value}` or concatenated. |
