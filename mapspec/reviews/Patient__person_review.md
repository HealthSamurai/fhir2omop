# Patient → person — review (vs OMOP CDM v5.4 spec)

Re-review of `Patient__person` (`mapspec/etl/Patient__person.sql`,
`mapspec/views/Patient__person.view.json`) against the authoritative
OMOP CDM v5.4 spec via `/omop-kg`, not against `refs/` implementations.
Only important findings.

## Critical

### 1. Patients without `birthDate` should be dropped, not partially loaded

`year_of_birth` field guidance:

> If no year of birth is available all the person's data should be
> dropped from the CDM instance.

(`person.year_of_birth.etl_conventions`, OMOP CDM v5.4.) The book ETL
chapter says the same in different words. `year_of_birth` is a
NOT NULL column.

We don't filter. `mapspec/etl/Patient__person.sql` calls
`EXTRACT(YEAR FROM v.birth_date::date)` unconditionally — a Patient
without `birthDate` (Synthea always has one, real EHR has gaps) would
either produce NULL and break the NOT NULL constraint, or fail the
cast and abort the whole stage. Either way the rest of that person's
clinical events would still flow into other tables with a
person_id pointing nowhere.

**Fix:** add `WHERE v.birth_date IS NOT NULL` to the stage-2 SQL.
Optionally surface a count via `script/validate-staging.ts` so the
operator sees how many patients were dropped.

### 2. Missing/unknown gender — OMOP book recommends drop; we keep

OMOP book "Extract Transform Load", Step 1 ETL design template:

> Drop any rows with missing/unknown gender. ... [8507/8532] were
> chosen as they are the only two standard concepts in the gender
> domain. The choice to drop patients with unknown genders tends to
> be site-based, though it is recommended they are removed as people
> without a gender are excluded from analyses.

Our `cm.gender_to_omop` maps `other → 8521` and `unknown → 8551`,
both **non-Standard** Athena Gender concepts. We keep the patient
in person.* with a non-Standard gender_concept_id.

This is an intentional, defensible divergence — preserves cohort
count and original-value lineage in `gender_source_value` — but it
diverges from book convention. Synthea has no `other`/`unknown`
gender so it doesn't trigger on our cohort.

**Action:** document the convention choice in the SQL header
"Intentional deviations" block. Either keep current (preserve
patients) or switch to 0 / drop. No code change required if we
explicitly state the position.

### 3. `care_site_id` ≠ patient's managingOrganization

`person.care_site_id` user_guidance:

> The Care Site refers to where the Provider typically provides the
> primary care.

That is, the GP's care_site — same row as the GP's
`provider.care_site_id`, set by `PractitionerRole__provider`.

We currently write `referenceToId(managing_organization_ref)` —
Patient.managingOrganization, which is the patient's primary
organization. On Synthea those are the same physical row (the GP
works at the org that is also the patient's primary org), so the
diff is invisible. On real EHR they can diverge: a patient is
managed by one organization while their GP works at another.

**Fix:** join through Practitioner→PractitionerRole→care_site.
Until that join is wired in, the current approximation is OK for
Synthea but the SQL header should note the semantic gap.

## Worth noting

### 4. `ethnicity_concept_id` is explicitly US-only

> Only use this field if you have US-based data and a source of this
> information. Do not attempt to infer Ethnicity from the race or
> ethnic background of the Person.

For US data sources only 38003563 (Hispanic) / 38003564 (Not Hispanic) /
0. We're correct on Synthea (US data). For non-US deployments this
column should be 0 — worth documenting in CLAUDE.md when we add
non-Synthea inputs.

### 5. `location_id` = "last known location"

> The location refers to the physical address of the person. This
> field should capture the **last known** location of the person.

We extract `(address.where(use='official') | address).first()`. For
sources with address history (`Address.period` populated), this picks
arbitrarily, not necessarily the most recent. Synthea emits one
address per Patient so no diff today.

**Fix when needed:** order by `address.period.end` (or `start`
descending) before `first()`. Or extract all and pick latest in SQL.

## Already correct vs spec

- `person_id` — autogenerated integer, source id preserved in
  `person_source_value` ✓ (we hash Patient.id; spec allows
  autogenerate-or-source-if-integer).
- `gender_concept_id` "biological sex at birth, not gender identity"
  → we COALESCE(us-core-birthsex, gender), preferring biological
  sex ✓.
- `gender_source_concept_id` — populated from `cm.gender_to_omop`
  with M/F → 8507/8532 (Athena Gender vocab) ✓.
- `birth_datetime` — COALESCE(birth_time::timestamp, birth_date::timestamp)
  picks up patient-birthTime extension ✓.
- `race_concept_id` REQUIRED, single concept (mixed races not
  supported) → COALESCE chain with explicit 0 fallback ✓.
- `*_source_concept_id` always 0 — spec says it "tends to be zero"
  for race/ethnicity; we honor this ✓.
- `person_source_value` = Patient.id (traceability key, not PHI) —
  matches user_guidance ✓.
