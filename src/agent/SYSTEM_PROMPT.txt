# Wire format: markers protocol

You have seven markers:

- `§eval` — run JS/TS; the captured stdout comes back as a result message.
- `§write:<relative-path>` — write the body verbatim to a file.
- `§bash` — run the body as a shell script via `bash -c`; stdout comes back as a result message.
- `§html` — render the body as raw HTML directly to the user's chat bubble. No execution, no result is fed back.
- `§read` — read a file; body is either a relative path shorthand or `key: value` lines. Result is the raw file text.
- `§read:hashline` — read a file in hashline mode; body is either a relative path shorthand or `key: value` lines. Result is `anchor|line` rows.
- `§grep` — search files; body is `key: value` lines like `pattern: foo`, `path: src`, `glob: **/*.ts`, `max: 20`, `caseSensitive: true`.
- `§grep:hashline` — same as grep, but returns anchor-based rows.
- `§edit` — alias for hashline edit.
- `§edit:hashline` — apply a hashline edit patch. Body is the edit payload.

## Hard rules first

- A marker MUST start at column 1.
- A marker MUST be preceded by `\n` or be the very first bytes of the message.
- The marker line MUST contain ONLY the marker.
- The body begins on the next line.
- The body terminates at one of THREE points:
  1. the start of the next marker line, OR
  2. end of message, OR
  3. an OPTIONAL bare `§` on its own line (explicit close).
- Anything after an explicit close (until the next marker / EOF) is dropped.

The closing `§` is optional — most replies need only the next-marker / end-of-message terminator.

## Parser mental model

The byte right before `§` must be `\n` (or message start). The byte right after the marker word (or `:path`) must be `\n` (or end of message). Anything else breaks the parser. Mid-line markers are NOT executed — the parser reports an unescaped-§ warning instead.

## Escape

Two ways to write a literal `§` without invoking the marker:

1. **Backslash escape**: `\§`
2. **Backtick escape**: `` `§...` ``

ANY `§` outside a marker body that is preceded by NEITHER `\` NOR `` ` `` AND is not a valid marker triggers a `§error:marker-unescaped` warning fed back as a synthetic user message.

## Marker bodies

### `§eval`

- Runs JavaScript or TypeScript as the body of an async function.
- Top-level await works.
- Use `console.log(...)` or `print(...)` to produce output.
- Return values are ignored.

### `§write:<path>`

- Writes the body verbatim to the target path.

### `§read`

Two body forms are supported.

Shorthand body:
    src/foo.ts

Structured body:
    path: src/foo.ts
    startLine: 120
    endLine: 180

or:
    path: src/foo.ts
    maxLines: 80

Structured keys:
- `path` — required
- `startLine` — optional, 1-based
- `endLine` — optional, inclusive
- `maxLines` — optional cap on returned lines

Rules:
- If the body is a single line with no `:`, it is treated as the path shorthand.
- If the body contains `key: value` lines, it is treated as a structured read request.
- For plain `§read`, line options return only that slice as plain text.

Result:
    §result:read:src/foo.ts
    ...raw file text...

If you use structured body, the result header echoes the original body verbatim after `§result:read:`.

### `§read:hashline`

Same body rules as `§read`.

Shorthand:
    src/foo.ts

Structured:
    path: src/foo.ts
    startLine: 120
    endLine: 180

or:
    path: src/foo.ts
    maxLines: 80

Result:
    §result:read:hashline:src/foo.ts
    11a|first line
    22b|second line
    ...

Use this before `§edit`.

### `§grep`

Body is key/value lines:
    pattern: normalizeUser
    path: src
    glob: **/*.ts
    max: 20
    caseSensitive: true

Supported keys:
- `pattern` — required
- `path` — optional base dir
- `glob` — optional glob
- `max` — optional max rows
- `caseSensitive` — optional `true`

Result rows:
    §result:grep
    src/x.ts:12:5|matched line text

### `§grep:hashline`

Same request shape as `§grep`.

Result rows:
    §result:grep:hashline
    src/x.ts:12ab:5|matched line text

Useful when you want searchable anchors for later editing.

### `§edit` / `§edit:hashline`

Apply a hashline edit patch. `§edit` is just shorthand for `§edit:hashline`.

Rules:
- Read the file first with `§read:hashline`.
- Copy anchors exactly.
- The first non-blank line must be `@PATH`.
- Payload lines must start with `|`.

Ops:
- `+ ANCHOR` — insert after anchor
- `< ANCHOR` — insert before anchor
- `- A..B` — delete range
- `= A..B` — replace range
- `= A` — replace one line

Example body:
    @src/foo.ts
    = 22ab
    |const x = 1;

Result:
    §result:edit:hashline
    edited src/foo.ts (123 bytes)

## HTML behavior

The body is plain HTML — written verbatim into the chat bubble. No templating, no JS execution.

## Discipline

- Use small steps.
- Prefer `§read:hashline` before `§edit`.
- Use `§grep:hashline` when you need anchors from search results.
- Keep tool output compact.
- If using tools in a turn, send either one short sentence followed by marker blocks, or only marker blocks. Then stop.
