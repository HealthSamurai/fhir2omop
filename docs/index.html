02 / THE REALIZATION

Agents need less than we ship them

A year in, a simple loop with tools is enough for programming work.

Mario Zechner's pi-mono made it concrete: one bash tool, ~1000 lines of harness, on par with Claude Code and Codex. A tenth of the size — and the gap isn't where you'd expect.

Why it works: agents already compose shell into pipelines and write code. So why ship a pre-baked toolbox when the agent can write the tools when it needs them, shaped to the user, the project, the task at hand?

03 / EMACS, AGAIN

Emacs was never an editor with plugins

It's a Lisp runtime that happens to edit text and rewrites itself on the fly.

That's why people are still building whole workflows inside it forty years later.

The same trick fits a coding agent: give it a runtime + a REPL, one ability — write and run code — and stop there. The rest (read, write, edit, grep, subagent, whatever comes next) the agent will write itself.

04 / WHY CODE

Let the agent think and act in code

Code is both — variables and control flow are thought; fetch, git, db calls are action. One medium for both.

MCP and bash give a flat menu of pre-shaped buttons; the agent has to verbalize plans in prose, then translate each step into a tool call. Code skips the translation. One evalCode can do the work of six MCP servers, with composition (loops, parallelism, libraries) for free.

Take a real fan-out: "triage every open issue in this repo and label it bug / feature / question."

const repo = "niquola/hyper-code2";
const issues = await (await fetch(
  `https://api.github.com/repos/${repo}/issues?state=open`,
  { headers: { Authorization: `Bearer ${process.env.GITHUB_TOKEN}` }})).json();

// fan out — one fork per issue, in parallel
const results = await Promise.all(issues.map(async (issue) => {
  const child = ctx.fns.session.fork(ctx, agent.id, agent.messages.length);
  await ctx.fns.agent.run(ctx, child,
    `Triage #${issue.number} "${issue.title}":\n${issue.body}\n` +
    `Reply with JSON: { label: "bug"|"feature"|"question", reason }.`);
  return { n: issue.number, ...JSON.parse(child.scratchpad.last) };
}));

agent.scratchpad.triage = results;
return { triaged: results.length, by_label: Map.groupBy(results, r => r.label) };

Related work: Recursive Language Models (Zhang/Kraska/Khattab, 2025) make the same architectural bet — long inputs are data inside a REPL environment, not text in the LLM context; the model writes code that peeks, slices, and recursively calls sub-LLMs over pieces. Our agent.scratchpad + ctx.fns.repl.eval + ctx.fns.session.fork give the agent the same shape today.

05 / WHY HYPERMEDIA

HTTP & HTML beat TUI and IDE plugins

The browser is a universal client we already have. The agent extends the UI the same way it writes any other code.

  • Add a screen instantly — drop a $route_*.ts file, it's live on the next request. No recompile, no extension reload, no manifest.
  • Push to the user — SSE channel, one ctx.fns.events.emit call → the browser reacts. No poll loop, no proprietary event API.
  • Real styling — Tailwind, shiki, CodeMirror, the entire web stack. Not ANSI escapes or an IDE theme API.
  • Renders before JS — server-side HTML is usable instantly. No cell-based redraw, no flicker, no first-paint dance.
  • Sharable — every state has a URL. Copy, paste, link, bookmark.
  • Multiplayer — many tabs, many users, same SSE channel. Free.
  • Inspectable — DevTools, view-source, network tab. Debugging is just web debugging.
  • Cross-platform — desktop, mobile, ssh-tunneled, embedded — anywhere with a browser.
  • Composable with the rest of the web — fetch, iframe, OAuth, file upload, drag-and-drop — all standard.

The mechanic in hyper-code2: agent writes .hyper/<mod>/$route_X_GET.ts, calls ctx.fns.http.loadRoutes(ctx), the route is live. Same trick for $script_* client JS, $type_* shared types, even Bun-bundled .entry.ts.

06 / SELF-AWARENESS

An agent that understands itself

Self-extension only works if the agent can see itself first.

In hyper-code2 the agent's whole world is a few obvious objects. It can read each one with the same evalCode it uses for anything else:

// What functions do I have?
Object.keys(ctx.fns)
// → ['agent', 'db', 'events', 'files', 'http', 'llm', 'markdown',
//    'repl', 'session', 'settings', 'ui', ...]

// What routes do I serve?
Object.fromEntries(Object.entries(ctx.routes).map(([p, m]) => [p, Object.keys(m)]))
// → { '/': ['GET'], '/agent/:id': ['GET','POST','DELETE'], ... }

// What is my own loop?
await Bun.file("src/agent/run.ts").text()

// What did I do an hour ago?
ctx.fns.db.select(ctx,
  "SELECT type, payload FROM events WHERE agent_id=? ORDER BY ts DESC LIMIT 20",
  [agent.id])

// Who am I?  →  agent.id, agent.model, agent.systemPrompt, agent.scratchpad

The system prompt teaches it the three places that matter: ctx.fns (what it can do), ctx.routes (what it exposes), SQLite (what it remembers). Once those are visible, "extending myself" becomes a concrete loop: see the gap → write the file → hot-load → use it. No reflection, no plugin API — the source code is the introspection surface.

07 / HYPER-CODE2

A runtime where the agent can live and extend itself

Small enough that the agent holds the architecture in context. Simple enough that it can extend it without breaking anything.

  • Bun — fast runtime, batteries-included (Bun.serve, Bun.file, Bun.$, bun:sqlite, Bun.build), ~zero deps.
  • SQLite for sessions — agents, messages, events, all queryable by the agent itself.
  • Web UI — SSR chat, files explorer (shiki + CodeMirror+vim), settings page (device-flow logins for Kimi-coding / Codex subscriptions, JWT auto-refresh).
  • About 1000 lines. Works with OpenAI, Anthropic, Groq, OpenRouter, LM Studio.

08 / REPL · PROCEDURAL DESIGN

Designed for REPL

Three rules that make hot-swap trivial.

  1. Functions take everything as parameters. No this, no closures over module state, no DI container. First arg is always ctx — anything the function needs comes through it.
  2. All state lives in one place: ctx. ctx.state.* in memory; SQLite for persistence; agent.scratchpad for off-LLM working memory. State never hides inside a module.
  3. Any function can be hot-swapped at runtime. Edit the file → ctx.fns.repl.load(ctx, "agent.run") → next call uses the new version. No restart, no re-init, no stale closures because there were no closures to begin with.
// All the data this function will ever touch comes through ctx + args:
export default async function (ctx: Context, path: string): Promise<string> {
    const abs = ctx.fns.files.resolveSafe(ctx, path);
    return Bun.file(abs).text();
}
//   →  loaded as ctx.fns.files.read   ·  hot-swap with one repl.load call

The agent uses the same REPL we do: bun script/repl.ts 'await ctx.fns.repl.load(ctx, "agent.run")' — and edits its own functions the same way.

No DI, no framework, no build step, no module-level state. In the spirit of Out of the Tar Pit.

09 / EXTEND IT TO YOUR PROJECT

The codebase grows a library shaped to you

Once. Twice. The third time the agent pins the pattern as a project procedure — and uses it as a built-in forever after.

Turn 1 — ad-hoc evalCode (one call replaces six MCP tools)

// "Pull release notes since the last tag — commits, authors, linked PRs."
const tag = (await Bun.$`git describe --tags --abbrev=0`.text()).trim();
const raw = await Bun.$`git log ${tag}..HEAD --pretty=format:%H|%s|%an`.text();
const cached = await ctx.fns.db.select(ctx,
  "SELECT body FROM release_notes WHERE tag = ? AND head = ?", [tag, head]);
if (cached.length) return { since: tag, cached: true };

const prs = await Promise.all(prNums.map(n =>
  fetch(`https://api.github.com/repos/${repo}/pulls/${n}`).then(r => r.json())));

agent.scratchpad.release = { tag, commits, prs, body };  // off-context
return { since: tag, commits: commits.length, contributors, prs_linked };

Turn N — pin the pattern as a project procedure

// Save under .hyper/ (gitignored, runtime-writable, agent-owned)
await Bun.write(".hyper/release/notes.ts", `
  export default async function (ctx: Context, repo: string) {
    // ... same logic — now a reusable function
    return { tag, commits, prs, body };
  }
`);

// Hot-load into the live runtime + regenerate types — no restart
await ctx.fns.repl.load(ctx, "release");
await ctx.genTypes(ctx);

// Turn N+1: call it like a built-in
await ctx.fns.release.notes(ctx, "niquola/hyper-code2");

Over weeks the project quietly grows a library of procedures shaped to this repo, this author, this team's habits. The agent learns your codebase, your conventions, your favourite shortcuts — and bakes them in. No fixed harness gives you that, because no fixed harness knows you.
